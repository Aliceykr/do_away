# Panel Issues Record

## Current issues

1. Only one panel should be visible at a time, but the generated button callbacks are independent.
`ui_event_wifiButton()` only opens `ui_WifiPanel`, and `ui_event_waveButton()` only opens `ui_wavePanel`, so cross-panel switching is not coordinated.

2. The generated animation names are opposite to their actual behavior.
In [My_LVGL/ui.c](C:/Users/Aliceykr/Desktop/do/do_away/My_LVGL/ui.c), `slideout_Animation()` moves `x` from off-screen to `0`, while `slidein_Animation()` moves `x` from `0` back off-screen.

3. Hidden positions are not consistent.
`ui_WifiPanel` starts at `x = 700`, while `ui_wavePanel` starts at `x = 800`, so a fixed "hide to 800" rule is not reliable for every panel.

4. Directly editing generated panel callbacks is fragile.
[My_LVGL/screens/ui_Screen1.c](C:/Users/Aliceykr/Desktop/do/do_away/My_LVGL/screens/ui_Screen1.c) is generated by SquareLine, so direct edits there are easy to lose after re-export.

5. Future panel growth needs a centralized switch path.
If more panels are added later, button-to-button `if` logic will become hard to maintain and easier to break.

## Current handling

1. Panel switching logic is placed in [APP/panelchange.c](C:/Users/Aliceykr/Desktop/do/do_away/APP/panelchange.c) and [APP/panelchange.h](C:/Users/Aliceykr/Desktop/do/do_away/APP/panelchange.h).

2. Runtime binding removes the generated open/close callbacks from the panel buttons and replaces them with centralized switching callbacks.

3. The initial `x` of each panel is treated as its hidden position, and `x = 0` is treated as its shown position.
