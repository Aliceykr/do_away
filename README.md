# STM32H750 RGB LCD + LVGL + WiFi + 波形控制工程

这是一个基于 `STM32H750XBH6` 的嵌入式工程，当前集成了：

- `FreeRTOS`
- `LVGL 8.4`
- `800x480 RGB LCD`
- `GT911` 触摸输入
- `UART4` WiFi 通信
- 波形页面 UI 与参数控制接口

## 当前分支说明

当前分支名称：

- `AD9833来提供波形输出方案`

这个分支的目标，不是继续沿用 `main` 分支里“STM32 片上 DAC 直接输出波形”的方案，而是把波形输出职责迁移到后续的 `AD9833` 外部波形发生器。

因此，本分支的工作重点是：

- 保留现有 `LVGL` 波形页面与交互接口
- 保留频率、`Vpp`、波形类型这些上层参数入口
- 移除片上 `DAC / TIM6 / DMA` 的出波逻辑
- 给后续接入 `AD9833` 预留干净、稳定的代码接口

一句话概括，本分支当前是：

`界面和参数控制先保留，片上 DAC 出波链已经撤掉，等待后续接入 AD9833 后端。`

## 本分支当前代码状态

- 波形页面上的 `SwitchA / SwitchB / SwitchC`、频率输入框、`Vpp` 输入框、`CHANGE` 按钮接口都继续保留
- `Wave_ApplyOutputToHardware()` 仍然保持为空，作为后续接入 `AD9833` 的唯一硬件出口
- 这次修改的目的不是恢复出波，而是把“参数层”和“旧 DAC 输出链”彻底拆开

- `APP/Wave.c` 与 `APP/Wave.h` 不再承担片上 DAC 出波任务
- 原先用于片上 DAC 的波形生成逻辑已经移除
- 原先 DAC DMA 相关回调空壳已经移除
- 原先 `DACStartTask` 独立线程入口已经移除
- 波形模块改为由 `defaultTask` 周期性调用 `Wave_Init()` 和 `Wave_Service()`
- 波形页面上的 `SwitchA / SwitchB / SwitchC` 接口保留
- 频率输入框接口保留
- `Vpp` 输入框接口保留
- `CHANGE` 按钮触发流程保留

- 原来为 DAC 波形缓冲单独预留的 `D2 SRAM` 区域已经移除
- 你已经在 CubeMX 中关闭了片上 `DAC / TIM6 / DMA` 这条旧出波链

需要特别说明的是：

- 当前分支现在不会再通过 STM32 片上 `DAC` 输出波形

- 所以当前状态是“参数层保留、硬件输出后端留空”

也就是说，现在这个分支的重点已经不是“出波”，而是“把原来和片上 DAC 紧耦合的上层控制逻辑拆开，为 AD9833 接管做准备”。

## 与 `main` 分支的核心区别

`main` 分支和当前分支最本质的差异，不在界面，而在波形输出链路。

### `main` 分支的思路

`main` 分支使用 STM32 片上 DAC 输出波形，典型路径是：

`LVGL 参数 -> Wave.c -> LUT / DDS / 采样数据 -> DMA -> DAC1 -> TIM6 触发输出`

它的特点是：

- 波形数据由 MCU 本地生成
- 使用 `DAC1 + TIM6 + DMA`
- 需要持续维护波形缓冲
- 需要相关 DAC 回调和任务配合
- 之前还为波形 DMA 缓冲预留了专门的 `D2 SRAM`

### 当前分支的思路

当前分支的路径已经改成：

`LVGL 参数 -> Wave.c -> 参数整理与状态同步 -> Wave_ApplyOutputToHardware() 占位 -> 预留 AD9833 接入`

它的特点是：

- 不再在 `Wave.c` 内生成片上 DAC 波形数据
- 不再保留原来的 LUT / DDS / DMA 流式出波逻辑
- 不再保留独立 `DACStartTask`
- 不再保留 DAC 回调链
- 上层页面接口继续保留，方便后续直接接入 `AD9833`

## 详细对比表

| 对比项 | `main` 分支 | 当前 `AD9833来提供波形输出方案` 分支 |
| --- | --- | --- |
| 波形输出硬件 | STM32 片上 `DAC1` | 计划迁移到 `AD9833` |
| 波形生成位置 | `Wave.c` 内部生成采样数据 | `Wave.c` 只保留参数处理与占位接口 |
| 定时触发 | `TIM6` | 当前已不再使用这条链路 |
| DMA | 使用 DAC DMA 流式搬运 | 已移除该逻辑 |
| 波形任务模型 | 独立 `DACStartTask` | 改为 `defaultTask` 中调用 `Wave_Service()` |
| DAC 回调 | 需要 | 已移除 |
| `D2 SRAM` 波形缓冲 | 有专门预留 | 已清理 |
| `LVGL` 波形页面 | 有 | 保留 |
| Switch 波形选择 | 有 | 保留 |
| 频率输入框 | 有 | 保留 |
| `Vpp` 输入框 | 有 | 保留 |
| `CHANGE` 按钮 | 有 | 保留 |
| 最终硬件输出 | 已由 DAC 实现 | `AD9833` 后端尚未实现，当前留空 |

## 当前 `Wave` 模块还保留了什么

为了保证你现有页面和交互逻辑尽量不返工，当前 `Wave` 模块保留了以下内容：

- 波形类型状态管理
- 频率参数解析
- `Vpp` 参数解析
- `CHANGE` 按钮触发后的参数应用流程
- 与 `LVGL` 控件的绑定逻辑
- 对外统一的波形控制接口
- 统一的硬件输出适配占位函数

这些保留项的意义是：

- 不破坏你当前的 `LVGL` 波形页面
- 不破坏你已经做好的人机交互流程
- 后面接入 `AD9833` 时，主要只需要补底层硬件输出部分
- 尽量避免以后因为更换输出硬件而反复改 UI 层

## 当前 `Wave` 模块已经去掉了什么

相对 `main` 分支，当前分支已经明确去掉了以下内容：

- 片上 DAC 波形生成逻辑
- LUT 查表出波逻辑
- DDS/NCO 相关的采样维护逻辑
- DMA 半缓冲 / 全缓冲续填逻辑
- 与 DAC 连续流式出波直接绑定的任务实现
- 与 DAC DMA 链路相关的回调函数
- 波形 DMA 专用 `D2 SRAM` 预留

这意味着当前分支已经不再沿用原先“MCU 本地造波 + DAC 实时推出去”的实现路线。

## 当前模块是怎么工作的

现在的 `APP/Wave.c` 主要做四件事：

1. 绑定并轮询波形页面上的 `LVGL` 控件
2. 读取 `SwitchA / SwitchB / SwitchC`，确定目标波形类型
3. 解析频率输入框和 `Vpp` 输入框内容
4. 把参数统一整理后交给 `Wave_ApplyOutputToHardware()`

其中：

- `Wave_Init()` 用于初始化波形模块状态
- `Wave_Service()` 由 `defaultTask` 周期性调用
- `Wave_ApplyOutputToHardware()` 当前仍为空，后续在这里接入 `AD9833`

这样做的好处是，上层界面层和下层具体输出芯片之间已经开始解耦。

## 当前状态下要注意的结论

当前代码状态要明确区分成两句话：

第一句：

`片上 DAC 出波链已经清理掉了。`

第二句：

`AD9833 的实际硬件输出逻辑还没写，所以当前分支还不会真正输出波形。`

这并不是漏改，而是当前分支的设计目标本来就是：

- 先拆掉旧 DAC 路线
- 再保留 UI 和参数层
- 最后单独补 AD9833 后端

## 后续接入 AD9833 的建议位置

后续如果你继续在这个分支上开发，建议主要在以下位置补代码：

- `APP/Wave.c`
  - 在 `Wave_ApplyOutputToHardware()` 中补 `AD9833` 配置与输出逻辑
- 如果后面需要独立的 `AD9833` 驱动文件
  - 建议新增独立驱动模块，不要再把旧 DAC 逻辑放回 `Wave.c`

推荐的开发顺序是：

1. 先完成 `AD9833` 的寄存器写入与基础驱动
2. 再把波形类型和频率参数接到 `Wave_ApplyOutputToHardware()`
3. 最后联调 `LVGL` 页面参数与实际输出结果

## 总结

如果只用一句话来描述当前分支和 `main` 分支的区别：

`main` 分支是“STM32 自己用片上 DAC 直接出波”，当前 `AD9833来提供波形输出方案` 分支是“保留原有界面和参数入口，移除片上 DAC 出波链，等待后续由 AD9833 接管波形输出”。`

所以，这个分支现在最重要的价值不是“已经能输出波形”，而是“已经把上层控制接口从旧 DAC 路线里剥离出来，后续接 AD9833 时改动面会更小、更干净”。
